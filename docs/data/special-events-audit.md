> `kind=special`(走過去按空白觸發)事件,對照已接清單 + 逐個查對話 record 文字(docs/script/txtNN.txt,
> bank=dlg_bank、rec=byte4),判讀每個 special 的真實性質,找出「攻略有、remake 未接」的真缺口。

## 方法與發現

- special 共 **53 個去重事件**(同 cty+dlg);已接 = scripted 表 b4 {7,12,16,25,31,44,49,50,52,84} +
  main.c 特例 {(14,58)甘達特巢穴,(15,25)古布達,(19,45)八頭大蛇,(82,64)不死鳥}。其餘 **40 個未接候選**。
- **關鍵發現**:逐個查對話文字後,40 個未接 special **絕大多數是「條件對話 NPC / 提示居民 / 設施說明」,
  不是 boss 或給道具缺口**。sub2 handler 反組譯(byte4=45/58)也證實:handler 是過場腳本
  (對話/NPC 動畫/flag/warp),**不含 [0x2321] 敵群表 / sub_bfd1 戰鬥呼叫 / 0x7bbe GIVE**——
  戰鬥與給道具是劇情強制或其他機制,非 sub2 handler 內。

## 結論

1. **真正的事件鏈缺口已接完**:道具獲得鏈(B-1~B-9)、boss(甘達特/八頭大蛇/巴拉摩斯序列)、
   過場(救人/彩虹/不死鳥)在前面 session 已接並 game_tester 驗證。
2. **剩餘 special 絕大多數是條件對話 NPC**,不是缺口——顯示對話即可,無需特殊接線。
   remake 既有 sub<2 對話型 NPC 路徑已能處理(`dq3_sub2_dialogue`)。
3. **方法論沉澱**:`list_special_events.py`(定位「哪一格觸發」)+ 對話文字判讀(定性質)+
   `dis_handler_full.py`(反組譯確認 handler 行為)= 可靠的「事件缺口」盤點三件組,不靠記憶/實機臆測。
4. 後續若要逐一接 CTY6 折扣 / CTY65 換蛋等長尾,屬精緻化,非可玩性缺口(主線已可破關)。
